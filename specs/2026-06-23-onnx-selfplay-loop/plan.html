★ What you're reviewing

The ask

Close the self-play loop on top of the existing data plane: (1) a Kotlin OnnxPolicy that runs a trained net inside the JVM to choose strategic actions (TECH + POLICY) from the legal mask, and (2) a Python training package + round driver that trains a small policy on emitted trajectories, exports ONNX, and measures win-rate vs RandomPolicy. The deliverable is a working end-to-end LOOP and a LEARNING CURVE — the go/no-go evidence for from-scratch self-play. Reuse the data plane; do not rebuild it.

How this plan solves it — TWO linchpins

Linchpin 1 — the policy must DRIVE the game, not just label it (caught by the plan council, FND-0002). Today NextTurnAutomation.onCivTurn is a recording hook: it records what chooseIndex would pick, but the engine's chooseTechToResearch/adoptPolicy heuristics make the real decisions. Training on those labels would REINFORCE non-causal actions, and "OnnxPolicy vs RandomPolicy" would secretly be heuristic-vs-heuristic (~50%, flat, for the wrong reason). The fix is a control seam (D14): the installed policy drives the learner's (and opponent's) tech+policy — pre-fill techsToResearch (the heuristic respects a non-empty queue) and adopt the chosen policy + guard the heuristic. Control runs in gen and eval; recorded action == applied action.

Linchpin 2 — the shards carry no learning signal. Reward is hardcoded 0f and there is no terminal flag. Wire a per-civ terminal ±1 reward into the shards (a sanctioned SampleSchema.VERSION 1→2 bump).

With both, REINFORCE-with-baseline trains a tiny MLP on concat(global, acting_civ) → {tech_logits, policy_logits}; the same net runs in-JVM via onnxruntime behind a RoutingPolicy (learner→ONNX, opponents→RandomPolicy). A plain-script driver loops generate→train→export→eval for K rounds, writes curve.csv + a plot, and emits an explicit GO / PLATEAU / INCONCLUSIVE verdict. A golden PARITY test pins the single observation code-path so train-time and infer-time never drift.

Architecture detail

The linchpin: terminal reward + VERSION 2 (current → proposed)

Current (DataPlaneHooks.recordCivTurn): writes turn, civSlot, isFirst, isLast=0, isTerminal=0, overflow, reward=0f + obs blocks + the actions block (4 chosen indices). Reward and terminal are placeholders — no learning signal.

Proposed: at game end, Simulation determines the per-civ outcome (win=+1 / lose=−1 / draw=0) and calls a new ShardRecorder.recordTerminal(...) that appends ONE record per recorded civ with isTerminal=1, reward=±1 (zero-filled obs — terminal obs is unused for training). SampleSchema.VERSION 1→2; Python SCHEMA_VERSION in lockstep (the reader already refuses a version mismatch → old shards correctly rejected, datasets are perishable by design). Python dataset.py groups steps by civ_slot, reads the terminal reward, and broadcasts it as the undiscounted return for that civ's non-terminal steps.

Win / tiebreaker definition (D7) — same for reward AND eval

Win = formal victory if one occurs. The turn cap defaults high so games play to a real victory (your decision). Only at the hard cap (the rare case) is the winner the civ with higher calculateTotalScore(); an exact tie is a draw (reward 0, excluded from the win numerator). Gated behind a new Simulation(scoreLeaderOnTimeout=true) flag so ConsoleLauncher/SimBenchmark behavior is unchanged.

Module placement (D4, revised by council + scan)

OnnxPolicy lives in desktop (where the onnxruntime native dep belongs), implementing core's PolicyProvider. RoutingPolicy + SimStats (no ORT dep) stay in core. onnxruntime is a desktop-only dependency (via gradle/libs.versions.toml). This keeps core ML-runtime-free — no dependency inversion, no compileOnly hack, zero Android-APK risk. Deviation from the task's "OnnxPolicy in core/.../dataplane" — see the Deviations table; the data plane already places heavy concerns (DataPlaneGen) in desktop.

ONNX I/O contract + anti-drift

Input tensor obs = concat(global, acting_civ) f32 (runtime GnK width). Outputs tech_logits[techCount], policy_logits[policyCount]. Widths read from the generated schema.json on both sides — never hardcoded (the task said policy=70; the scan estimates ~80; the truth is vocab.policyCount at runtime). policy.onnx carries schema_version, ruleset_fingerprint, contract_version as ONNX metadata; the JVM reads them via session.getMetadata().getCustomMetadata() and EVAL refuses a contract-mismatched model. The PARITY test feeds ONE fixed obs through the JVM session and a Python reference, asserting logits match within atol=1e-4.

# Plan — ONNX-in-JVM policy bridge + minimal self-play training loop

**Mode**: BUILD · **Size**: L · **Base**: `self-play-data-plane` · **PR target**: `self-play-data-plane`

## The ask
Close the self-play loop on top of the existing data plane: a JVM `OnnxPolicy` that runs a trained net **inside the JVM** to choose TECH + POLICY from the legal mask, plus headless GENERATE/EVAL entrypoints and a Python `unciv_train` package + round driver that trains on emitted trajectories, exports ONNX, and measures win-rate vs RandomPolicy. **The deliverable is a working end-to-end LOOP and an honest LEARNING CURVE** — the go/no-go evidence for from-scratch self-play. Reuse the data plane; don't rebuild it.

## How this plan solves it
There are **two linchpins** (the second was caught by the plan council, FND-0002):

1. **The policy must actually DRIVE the game, not just label it (D14, new P0).** Today `NextTurnAutomation.onCivTurn` is a *recording* hook — it records what `chooseIndex` would pick, but the engine's `chooseTechToResearch`/`adoptPolicy` heuristics make the real decisions. Training on those labels would REINFORCE non-causal actions, and "OnnxPolicy vs RandomPolicy" would secretly be heuristic-vs-heuristic (~50%, flat, wrong reason). The fix: a **control seam** so the installed policy drives the learner's (and opponent's) tech+policy — pre-fill `techsToResearch` (the heuristic respects a non-empty queue) and adopt the chosen policy + guard the heuristic. Control runs in gen **and** eval; recorded action == applied action.
2. **The shards record no reward (hardcoded 0) and no terminal flag** → no learning signal. Wire a per-civ **terminal ±1 reward** into the shards (a sanctioned `SampleSchema.VERSION` 1→2 bump).

With both, REINFORCE-with-baseline trains a tiny MLP on `concat(global, acting_civ)` → `{tech_logits, policy_logits}`; the same net runs in-JVM via onnxruntime through a `RoutingPolicy` that sends only the learner civ to ONNX (opponents stay RandomPolicy). A plain-script driver loops generate→train→export→eval for K rounds, writes `curve.csv` + a plot, and emits an explicit **GO / PLATEAU / INCONCLUSIVE** verdict. A golden PARITY test pins the one observation code-path so train-time and infer-time never drift.

## Why (rationale)
- **Reuse over rebuild**: every seam (Featurizer, masks, emitter, reader, Simulation harness, `binomialTest`) already exists and is verified. We add the *missing* reward signal and the *new* policy/runner/training pieces only.
- **One observation path**: `concat(global, acting_civ)` is produced identically on JVM (`Featurizer.observe`) and Python (`step.blocks`), golden-tested — the #1 risk (drift) is closed by construction.
- **Stationary opponent**: learner-vs-RandomPolicy gives a clean, low-variance learning signal for a true go/no-go (per the task's non-goals — no co-learning yet).

## Architecture: the loop
```
            ┌────────────────────────── run_loop.py (driver, K rounds) ──────────────────────────┐
 round 0:   │  gen (RandomPolicy)                                                                 │
 round r≥1: │  gen ──► ./gradlew selfPlay --args="gen <model> <out_r> ..."  → shards (VERSION 2)  │
            │   │                                                                                 │
            │   ▼                                                                                 │
            │  train ── unciv_train.dataset(shards) → REINFORCE-w-baseline → state_dict           │
            │   │                                                                                 │
            │   ▼                                                                                 │
            │  export ── export_onnx → policy.onnx  (fixed names + metadata: ver, fingerprint)    │
            │   │                                                                                 │
            │   ▼                                                                                 │
            │  eval ── ./gradlew selfPlay --args="eval policy.onnx <M> ..." → EVAL_RESULT {json}  │
            │   │                                                                                 │
            │   ▼  append curve.csv (round, games, winrate, pval) ; replot curve.png              │
            └─────────────────────────────────────────────────────────────────────────────────────┘
```
Inside the JVM (gen + eval): `Simulation` runs N/M Tiny 2-civ GnK games; `NextTurnAutomation.onCivTurn` → the existing single-policy `install()` → a `RoutingPolicy` that dispatches the **learner** civ to `OnnxPolicy` and **opponents** to `RandomPolicy`. `OnnxPolicy` builds `concat(global, acting_civ)` via `Featurizer`, runs one ONNX forward per (civ,turn), masks illegal logits to −inf, then samples (gen) or argmaxes (eval).

---

## P0 / P1 split
**P0 — required for a *meaningful* loop + all 6 acceptance criteria**
- **Control wiring (D14)**: the installed policy drives tech+policy for controlled civs (decision-gated; recorded==applied; gen+eval); `adoptPolicy` guard. *Without this the loop cannot learn.*
- Schema VERSION 2 + per-civ terminal reward (Kotlin emitter + Simulation hook; Python `SCHEMA_VERSION` lockstep).
- `RoutingPolicy` (core), `OnnxPolicy` (desktop: ThreadLocal per-(game,civ,turn) memo, masked sample/argmax, per-game error isolation), `SimStats` (core, extracted `binomialTest`).
- `SelfPlayRunner` (desktop: `gen`/`eval`/`parity-dump` modes) + gradle `selfPlay` task + onnxruntime dep.
- `unciv_train`: `contract`, `model`, `dataset` (provenance gate; per-head masking; group by (shard,civ_slot)), `train` (no negative-index gather), `export_onnx`, `run_loop` (subprocess arg-list + returncode/timeout checks; GO/PLATEAU/INCONCLUSIVE verdict; per-round shard retention).
- Tests: legality, determinism, parity, provenance, concurrent-eval smoke. `curve.csv` + plot + verdict.

**P1 — polish (do if cheap; never blocks the curve)**
- Structured per-round log line beyond curve.csv; plot styling; optional `γ` discount knob (default 1.0); entropy-bonus knob. *(Adaptive eval-stop, nation randomization, co-learning, heads beyond tech+policy = OUT, v2.)*

| 3. ONNX forward (memoized for civ42) | `OrtSession.run({"obs": [1,232]})` → `tech_logits[80]`, `policy_logits[~80]` | tech_logits idx7=2.1 (max legal) |
| 4. Mask + choose + **APPLY** | illegal→−inf; sample over {3,7,9}; **set `Babylon.tech.techsToResearch=[tech7]`** (control) + record `actions[0]=7` | applied==recorded; Babylon now researches the net's pick (not the heuristic's) |
| 5. Game ends (turn 188, Babylon domination win) | `Simulation` → `recorder.recordTerminal({Babylon:+1, Sumeria:−1})` | terminal step `civ_slot=0, reward=+1` |
| 6. Train (next round) | `dataset`: Babylon's step42 gets `R=+1`; masked logp(tech=7); `A=+1−baseline` | loss `−A·logp` → net nudges tech7 up |

# Decisions — ONNX policy bridge + self-play loop

## Fork / fork-in-order (pre-feature)
- Base branch = `self-play-data-plane` (the only branch with the data plane; backed up on `fork`). `master` left clean as the upstream (`origin` = yairm210/Unciv) mirror so an upstream PR stays clean later.

## Phase-1 user decisions (Step 3)
- **D-win**: Win = formal victory; raise the turn cap so games play to a real victory; score-leader only as an extreme-case tiebreaker at a hard cap. SAME definition for training reward (±1) and EVAL win-rate.
- **D-budget**: All loop knobs (K rounds, gen-games, eval-games, turn-cap, threads, seed, learner nation, eval/sample mode) are CLI+config-configurable with balanced defaults chosen by Claude.

## Default-rigor design decisions (Claude, technical)
- **D1 Terminal reward + VERSION bump**: shards record no reward/terminal today (hardcoded 0). Add per-civ terminal ±1 at game end; bump `SampleSchema.VERSION` 1→2; lockstep `python schema.py SCHEMA_VERSION=2`. Reader refuses old shards → regenerate each round (perishable-dataset design). Sanctioned by the task's NON-GOAL escape hatch.
- **D2 RoutingPolicy**: injection is one-policy-per-run; wrap with `RoutingPolicy(learnerId, onnx, random)` that routes by civ — no data-plane modification. Pin fixed nations in the Tiny config so learner identity (and `numWins[learner]`) is stable across games.
- **D3 SimStats extraction**: extract `binomialTest`/`normalCdf`/`erf` from `Simulation` into a public `SimStats` util for EVAL reuse + unit-testing; EVAL prints one machine-readable result line.
- **D4 onnxruntime placement** (REVISED after Step 5 council FND-0002/0006 + Step 7 scan): **OnnxPolicy lives in `desktop`** (where the ORT native dep belongs), implementing `core`'s `PolicyProvider`. `RoutingPolicy` + `SimStats` (no ORT dep) stay in `core/.../dataplane` + `core/.../simulation`. onnxruntime = a `desktop`-only `implementation` dep (added to `gradle/libs.versions.toml`). This keeps `core` ML-runtime-free (no dependency inversion, no compileOnly hack, zero Android risk) — the data plane already places heavy concerns (DataPlaneGen) in desktop. **Deviation from the task's "OnnxPolicy in core/.../dataplane" — justified; surfaced in the plan Deviations table.**
- **D6 Per-(civ,turn) inference cache** (council FND-0037): `chooseIndex` is called once per head (tech, then policy) per civ-turn → OnnxPolicy runs ONE forward pass per (civID, turn), memoizing both heads' logits; the 2nd head reuses it. Memo holds only the current (civ,turn) (tiny, cleared on key change).
- **D7 Win/tiebreaker (precise)** (council FND-0023/0027/0009): win = formal victory; else (only at the hard turn cap) the civ with higher `calculateTotalScore()`; exact tie = draw (reward 0, excluded from the win numerator). SAME for training reward + EVAL. Turn cap defaults high so the tiebreaker is the rare path (user decision).
- **D8 Lean surface** (council FND-0001/0003/0005/0004): ~6 driver knobs (K, gen-games, eval-games, turn-cap, threads, seed) with baked balanced defaults (the default operating contract); plain-script driver (no framework); hardcode the 2 heads (no generic head abstraction); `SimStats` = just the binomial test.
- **D9 Error handling + native memory** (council FND-0007/0022/0025): close `OnnxTensor`/`OrtSession.Result`; one shared session closed at run end; model-load/inference failure → fail fast (loud), never silent-fallback; model path is operator-supplied + self-produced (trusted).
- **D10 JVM↔Python contract** (council FND-0008/0021): driver invokes `./gradlew selfPlayGen|selfPlayEval` with explicit args; EVAL prints exactly one line `EVAL_RESULT <json>` (games, wins, winrate, pval, learner, seed, onnx_decisions) the driver parses; routing-held assertion (`onnx_decisions>0`).
- **D11 Documented v1 limitations** (council FND-0030/0033/0029): fixed-nation matchup → matchup-overfitting (acceptable for a stationary-opponent go/no-go signal; map seed varies per game; v2 = randomize nations); undiscounted return → high variance (baseline mitigates; γ optional knob, default 1.0); no opponent co-learning (v1 = learner-vs-Random, per task non-goal).
- **D12 Loop hygiene** (council FND-0011/0039/0013): turn cap bounds every game (no infinite hang) + a generous per-process timeout backstop in the driver; fresh per-round shard dir with retention (keep last N, configurable); VERSION-bump rollback = bump back + regen (perishable-by-design).
- **D13 Parity tolerance** (council FND-0024): PARITY test asserts logits match within `atol=1e-4` (f32).
- **D5 ONNX I/O contract**: input = concat(`global`,`acting_civ`) at the runtime GnK width; outputs `tech_logits[techCount]`, `policy_logits[policyCount]`. Widths read from vocab/schema (NOT hardcoded 80/70). VERSION + ruleset fingerprint stamped as ONNX metadata + a shared constant. PARITY test = JVM session vs Python reference on a fixed observation.

## Plan-council decisions (Step 11, from 42 findings)
- **D14 CONTROL wiring (THE fix, new P0)**: the installed policy must DRIVE tech+policy for the civs it controls, not just record labels (`onCivTurn` is recording-only today; the heuristic `chooseTechToResearch`/`adoptPolicy` make the real decisions). Design: `DataPlaneHooks` splits per-civ-turn handling into **controlApply** (always — choose+apply) and **emit** (gen only). Decision-gated: record/apply a TECH action only when `civInfo.tech.techsToResearch.isEmpty()` (pre-fill the queue → `chooseTechToResearch` respects it, line 276); record/apply a POLICY action only when `civInfo.policies.canAdoptPolicy()` (adopt the chosen policy); else record −1 (no decision this turn). `chooseIndex` is called ONCE per head per decision-turn → recorded action == applied action. Guard `NextTurnAutomation.adoptPolicy` to skip the heuristic loop for controlled civs. Control runs in BOTH gen + eval (eval installs the policy without registering a recorder). greatPerson/diplomaticVote stay heuristic (recorded labels only; not trained). Symmetric: learner = net for tech/policy + heuristic rest; opponent = random for tech/policy + heuristic rest → the learning signal is isolated to tech+policy.
- **D15 ThreadLocal memo**: OnnxPolicy's per-(civ,turn) forward-pass cache is a `ThreadLocal` keyed by (gameId, civID, turn) holding only the most recent — fixes concurrent-game corruption (FND-0001/0026/0023).
- **D16 Loss correctness**: per-head masking — skip a head's term when its recorded action `< 0`; assert `a>=0` before gather (no negative-index bug); keep a step if ANY head acted (don't drop whole steps) (FND-0024/0025).
- **D17 Go/no-go rule**: GO if final-round winrate ≥60% AND p<0.05; PLATEAU if winrate ∈ [45,55]% with no upward trend over the last ≥3 rounds; INCONCLUSIVE otherwise. The driver emits a verdict line alongside curve.csv (FND-0010/0018).
- **D18 Parity fixture**: `selfPlay parity-dump <seed>` writes a committed fixture observation; JVM-parity and Python-parity both read the SAME fixture; assert logits within 1e-4 (FND-0015).
- **D19 Driver/runtime robustness**: subprocess as an **arg-list** (no shell, no string interpolation); check returncode + per-call timeout; a failed round aborts loudly. OnnxPolicy inference error → abort THAT game (isolation), not the whole run. Validate model path exists (FND-0005/0006/0017/0019/0020/0021). Local-trust context: driver + JVM are the same trusted user.
- **D20 Concrete defaults + budget**: K=10, gen-games=24, eval-games=100 (60% vs 50% ⇒ p≈0.023), turn-cap=325 (high, per the win-rule decision), threads=cores−1, ORT intra-op=1. gen→train→eval are SEQUENTIAL phases ⇒ no JVM/ORT/torch thread contention. Hard total-games budget cap. Shard retention (keep-last-N) is P0 (per-round regen fills disk). dataset groups by (shard, civ_slot); training sorts shards by name (FND-0011/0033/0036/0007/0035/0027/0004/0034/0039).
- **D21 Responsible-use note (not engineered)**: a tech/policy picker for a turn-based strategy game has negligible real-world harm/dual-use; plan adds a one-paragraph scope note, no harm-reporting machinery (FND-0028/0029/0030/0031/0032 — over-reach for a game AI).
