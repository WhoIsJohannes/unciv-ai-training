The ask

Close the self-play loop on top of the existing data plane: (1) a Kotlin OnnxPolicy that runs a trained net inside the JVM to choose strategic actions (TECH + POLICY) from the legal mask, and (2) a Python training package + round driver that trains a small policy on emitted trajectories, exports ONNX, and measures win-rate vs RandomPolicy. The deliverable is a working end-to-end LOOP and a LEARNING CURVE — the go/no-go evidence for from-scratch self-play. Reuse the data plane; do not rebuild it.

How this plan solves it — TWO linchpins

Linchpin 1 — the policy must DRIVE the game, not just label it (caught by the plan council, FND-0002). Today NextTurnAutomation.onCivTurn is a recording hook: it records what chooseIndex would pick, but the engine's chooseTechToResearch/adoptPolicy heuristics make the real decisions. Training on those labels would REINFORCE non-causal actions, and "OnnxPolicy vs RandomPolicy" would secretly be heuristic-vs-heuristic (~50%, flat, for the wrong reason). The fix is a control seam (D14): the installed policy drives the learner's (and opponent's) tech+policy — pre-fill techsToResearch (the heuristic respects a non-empty queue) and adopt the chosen policy + guard the heuristic. Control runs in gen and eval; recorded action == applied action.

Linchpin 2 — the shards carry no learning signal. Reward is hardcoded 0f and there is no terminal flag. Wire a per-civ terminal ±1 reward into the shards (a sanctioned SampleSchema.VERSION 1→2 bump).

With both, REINFORCE-with-baseline trains a tiny MLP on concat(global, acting_civ) → {tech_logits, policy_logits}; the same net runs in-JVM via onnxruntime behind a RoutingPolicy (learner→ONNX, opponents→RandomPolicy). A plain-script driver loops generate→train→export→eval for K rounds, writes curve.csv + a plot, and emits an explicit GO / PLATEAU / INCONCLUSIVE verdict. A golden PARITY test pins the single observation code-path so train-time and infer-time never drift.

Architecture detail

The linchpin: terminal reward + VERSION 2 (current → proposed)

Current (DataPlaneHooks.recordCivTurn): writes turn, civSlot, isFirst, isLast=0, isTerminal=0, overflow, reward=0f + obs blocks + the actions block (4 chosen indices). Reward and terminal are placeholders — no learning signal.

Proposed: at game end, Simulation determines the per-civ outcome (win=+1 / lose=−1 / draw=0) and calls a new ShardRecorder.recordTerminal(...) that appends ONE record per recorded civ with isTerminal=1, reward=±1 (zero-filled obs — terminal obs is unused for training). SampleSchema.VERSION 1→2; Python SCHEMA_VERSION in lockstep (the reader already refuses a version mismatch → old shards correctly rejected, datasets are perishable by design). Python dataset.py groups steps by civ_slot, reads the terminal reward, and broadcasts it as the undiscounted return for that civ's non-terminal steps.

Win / tiebreaker definition (D7) — same for reward AND eval

Win = formal victory if one occurs. The turn cap defaults high so games play to a real victory (your decision). Only at the hard cap (the rare case) is the winner the civ with higher calculateTotalScore(); an exact tie is a draw (reward 0, excluded from the win numerator). Gated behind a new Simulation(scoreLeaderOnTimeout=true) flag so ConsoleLauncher/SimBenchmark behavior is unchanged.

Module placement (D4, revised by council + scan)

OnnxPolicy lives in desktop (where the onnxruntime native dep belongs), implementing core's PolicyProvider. RoutingPolicy + SimStats (no ORT dep) stay in core. onnxruntime is a desktop-only dependency (via gradle/libs.versions.toml). This keeps core ML-runtime-free — no dependency inversion, no compileOnly hack, zero Android-APK risk. Deviation from the task's "OnnxPolicy in core/.../dataplane" — see the Deviations table; the data plane already places heavy concerns (DataPlaneGen) in desktop.

ONNX I/O contract + anti-drift

Input tensor obs = concat(global, acting_civ) f32 (runtime GnK width). Outputs tech_logits[techCount], policy_logits[policyCount]. Widths read from the generated schema.json on both sides — never hardcoded (the task said policy=70; the scan estimates ~80; the truth is vocab.policyCount at runtime). policy.onnx carries schema_version, ruleset_fingerprint, contract_version as ONNX metadata; the JVM reads them via session.getMetadata().getCustomMetadata() and EVAL refuses a contract-mismatched model. The PARITY test feeds ONE fixed obs through the JVM session and a Python reference, asserting logits match within atol=1e-4.
