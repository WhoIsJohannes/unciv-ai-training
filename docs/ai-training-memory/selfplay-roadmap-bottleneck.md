The ONNX self-play loop's v2 (landed on `master`, branch `onnx-selfplay-loop` fast-forwarded to it 2026-06-24) already implements AND evaluated a board-reading "rich" representation (`RichPolicyValueNet` + `features.py` masked mean+max pooling + contract-v2 multi-tensor ONNX bridge + rich parity test + `--variant rich-critic`). Results in `specs/2026-06-23-onnx-selfplay-v2/results/`: rich beats blind on **Tiny** (57% vs 35% last-4 mean) but is significantly **worse on Medium** (14.7% vs 28.9%, z=−3.48), and rich's Medium curve *declines* round-over-round.

A proposed "v3: simple board-reading model" was assessed as **redundant** — the question "does seeing the map beat blind?" is already answered (yes on Tiny, no on Medium).

**Why:** v2's diagnosis is that rich is *undertrained* on Medium, not that the representation is bad — ~1261 spatial tiles (vs 331 Tiny), thinner budget, and **from-scratch training every round (no weight carryover)** caps how much a larger model can learn. Adding MORE representation capacity (a v4 GNN/attention encoder) before fixing this would make Medium *worse*, not better.

**How to apply:** the highest-leverage next feature is **weight carryover / continual training across rounds** (warm-start each round from the prior net), plus cheap complements (observation normalizer; entropy-coef/epochs tuning to fix entropy collapse). Treat that as a precondition before any richer encoder. See [[onnx-selfplay-v2-results]].

**Decision 2026-06-24:** when starting the v4 structured-encoder feature (`/feature`), I surfaced this diagnosis as a conflict (v4 adds encoder capacity but leaves from-scratch-per-round untouched). The user reviewed it and explicitly chose to **proceed with v4 as written** — building the GNN/embeddings/attention encoder under the existing per-round training regime, accepting the undertraining/attribution risk. So this caution has been acknowledged-and-overridden for v4; do NOT re-flag it on the v4 branch. Weight-carryover remains the recommended follow-up if v4's Medium result is ambiguous.

**v4 OUTCOME 2026-06-25 (built + measured; branch `onnx-selfplay-loop`, see `specs/2026-06-24-onnx-selfplay-v4-structured-encoder/RESULTS.md`):** the structured **hex-GNN** encoder (small/GNN-only rung) **fixes the Medium regression**: 200-game final **23.0% vs v3 rich-pool 14.7%, z=+2.20, p=0.014 (AC1 PASS)**, and the Medium curve no longer *collapses* (v3 rich declined 35→10%; GNN oscillates ~25% with no downtrend). BUT structured is **still below the blind-critic's 28.9%** (−5.9pp, not significant). Net: the masked-pool→GNN swap was a real fix (locality matters), yet the encoder **still can't beat blind** — which **confirms this diagnosis**: from-scratch-per-round training is the ceiling, not the representation. So the roadmap stands — **weight carryover / continual training is now the clearly-indicated next unlock**. The v4 encoder is done + parity-tested; it is NOT the bottleneck.

**v5 OUTCOME 2026-06-25 — BOTTLENECK RESOLVED (shipped to `master` 04f2e27fa via local FF; `specs/2026-06-25-selfplay-v5/RESULTS.md`):** continual training (warm-start net + carried Adam optimizer state across rounds, 16 rounds vs v4's 8) **clears the blind baseline for the first time**. 200-game Medium ceiling, seed 4242424: small rung **40.7% (83/204)** beats v4 from-scratch 23.0% (z=3.83, p=6.5e-05) AND clears blind 28.9% (z=2.46, **p=0.0069**); medium rung (now trainable on Medium via micro-batching, no OOM) is the strongest at **46.6% (95/204)**. Zero divergence over 32 rounds. **Key nuance:** continual wins by *accumulating* across rounds 8–15 (which from-scratch structurally can't reach), NOT by being better per-round early — at v4's matched 8-round mark v5 was ~19%, below v4's 23%. So "run longer under a persistent net" is what unlocks it. The encoder-vs-training-regime question is now fully settled: **the regime was the ceiling, and continual training broke it.** Next unlocks (out of v5 scope): (1) cross-round **replay buffer** — DEFERRED, requires recording behavior-policy log-probs in the Kotlin recorder + shard schema first (currently only the chosen int index is stored; PPO `old_logp` is recomputed in-round, valid only on-policy); (2) **modeled action heads** for units/cities/great-person/diplomatic-vote (still heuristic).

**v6 CHOSEN 2026-06-25 — replay buffer (in progress).** User picked the replay path over control-heads (foundation/sample-efficiency before the bigger capability leap). Design being built: (a) record per-head behavior log-probs in `OnnxPolicy` at the masked-softmax sample point + a new `behavior_logp` shard block, bumping shard schema VERSION 3→4 (reader/dataset in lockstep; the ONNX *model* export contract stays unchanged); (b) PPO uses the STORED behavior logp as `old_logp` for replayed off-policy data — a deliberate, scoped relaxation of the v5-frozen `train.py:144-147` recompute, rest of the update verbatim, GAE recomputed under the current value net; (c) small recent-round-window replay buffer (`--replay-window K`, default small e.g. 4) in `run_loop`, train on current ∪ last K-1 rounds; (d) no-op safety: `--replay-window 1` must reproduce v5 within fp tolerance. Goal: reach v5's 40.7% Medium ceiling in FEWER rounds, or a HIGHER ceiling at equal rounds. After replay: control heads, then self-play/league.
